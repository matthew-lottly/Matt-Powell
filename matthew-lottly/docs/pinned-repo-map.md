# Pinned Repo Map

Use this file to decide which private projects are worth promoting into public showcase repos.

## Ideal Pinned Set

### 1. Flagship integrated project

Purpose: make the account feel like a coherent engineering portfolio instead of four separate interests.

Best fit:

- A project that combines backend code, a database, and either GIS or analytics
- A system with inputs, processing, storage, and a usable output

Signals:

- Architecture thinking
- Practical engineering decisions
- Ability to connect multiple technical disciplines

### 2. Backend or database system

Purpose: show depth in software engineering and data architecture.

Best fit:

- API service
- ETL pipeline
- Postgres or PostGIS-backed application
- Schema and migration-driven project

Signals:

- Maintainable code
- Data modeling
- Operational reliability

### 3. GIS or geospatial application

Purpose: show that geospatial work is a serious capability, not just a side interest.

Best fit:

- Spatial analysis workflow
- Geoprocessing tool
- Map-centric application
- GIS automation project

Signals:

- Domain depth
- Spatial reasoning
- Tooling fluency

### 4. Analytics or data science project

Purpose: show analytical value and communication, not just notebooks.

Best fit:

- Reproducible analysis with strong outputs
- Forecasting, classification, or optimization project
- Reporting workflow with clear business or operational impact

Signals:

- Problem framing
- Interpretation
- Communicating results clearly

### 5. Optional developer tool

Purpose: show clean engineering habits and reusable code.

Best fit:

- CLI utility
- Python package
- Reusable library or automation tool

Signals:

- Code quality
- Reuse
- Good interfaces and documentation

## Promotion Rules

- Make a repo public only if the README is strong and the project has a clear purpose.
- Avoid exposing anything tied to private employer data, internal names, or restricted workflows.
- Prefer one polished public repo every few months over dumping many mediocre repos at once.
- If a project is good but too sensitive, extract a safe reusable component and publish that instead.

## Quick Scoring

Score each candidate repo from 1 to 5 in these areas:

- Technical depth
- Documentation quality
- Professional safety for public release
- Visual or demonstrable output
- Fit with your overall profile story

Projects scoring well across all five areas should be first in line for public release.
